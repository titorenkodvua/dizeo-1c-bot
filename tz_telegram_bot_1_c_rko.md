# ТЗ на разработку Telegram-бота для работы с 1С

## 1. Цель
Разработать Telegram-бота, который работает как простой интерфейс к HTTP-сервису 1С.

### Версия v1 должна уметь только:
1. Показывать список последних расходно-кассовых ордеров.
2. Создавать новый расходно-кассовый ордер.

Бот нужен для личного/внутреннего использования, без публичного доступа.

---

## 2. Основной сценарий работы

### Функция 1. Просмотр списка РКО
Пользователь нажимает кнопку **«Показать РКО»**.  
Бот обращается к HTTP-сервису 1С, получает список документов и выводит его в Telegram в читаемом виде.

### Функция 2. Создание РКО
Пользователь нажимает кнопку **«Создать РКО»** и вводит сумму и комментарий.  
Бот валидирует сумму, показывает подтверждение и только после подтверждения создаёт документ в 1С.

### Функция 3. Быстрый ввод одной строкой
Если пользователь отправляет сообщение вида:

- `1000 изолента`
- `250,50 саморезы`
- `1223 доставка`

бот должен распознать это как команду на создание РКО, выделить сумму и комментарий, показать подтверждение и после подтверждения отправить запрос в 1С.

## 5. API 1С

### 5.1. Базовый URL
Использовать базовые настройки:

- `ONEC_BASE_URL`
- `ONEC_API_PATH`

Пример:
```text
ONEC_BASE_URL=http://1cbot.tdvkrakow.keenetic.link
ONEC_API_PATH=/1capi/hs/botapi
```

Итоговый endpoint:
```text
http://1cbot.tdvkrakow.keenetic.link/1capi/hs/botapi/cash-expense-orders
```

### 5.2. Авторизация к 1С
Авторизация должна храниться **отдельно в env**, а не внутри URL.

Использовать Basic Auth:

- `ONEC_USERNAME`
- `ONEC_PASSWORD`

Пример:
```text
ONEC_USERNAME=admin
ONEC_PASSWORD=optiplex
```

В коде нельзя использовать URL вида:
```text
http://admin:password@host/...
```

### 5.3. Получение списка РКО
**Метод:** `GET`  
**Endpoint:** `/cash-expense-orders`  
**Query params:**
- `limit`

Пример:
```http
GET /cash-expense-orders?limit=10
```

**Фактический формат ответа 1С:**
```json
{
  "items": [
    {
      "number": "TY000000017",
      "date": "29.03.2026 0:19:42",
      "posted": true,
      "cashbox": "Касса нал",
      "expense_item": "Прочие",
      "counterparty": "",
      "comment": "тратата",
      "sum": 1223
    }
  ]
}
```

**Расшифровка полей:**
- `number` — номер документа
- `date` — дата документа
- `posted` — признак проведения документа
- `cashbox` — название кассы
- `expense_item` — статья затрат
- `counterparty` — контрагент, если есть
- `comment` — комментарий
- `sum` — сумма

**Ожидаемое поведение:**
- бот передаёт `limit`
- получает JSON
- читает массив `items`
- если `items` пустой, показывает сообщение: `РКО не найдены`
- если элементы есть, выводит их в удобном читаемом виде

**Формат вывода в Telegram:**
Для каждого документа выводить отдельный блок, например:
```text
✅ TY000000017
Дата: 29.03.2026 0:19:42
Сумма: 1223
Касса: Касса нал
Статья: Прочие
Контрагент: —
Комментарий: тратата
```

Правила отображения:
- `posted=true` → показывать `✅`
- `posted=false` → показывать `❌`
- если `counterparty` пустой, показывать `—`
- если `comment` пустой, показывать `—`
- документы выводить в порядке, в котором они пришли от 1С
- бот не должен падать, если какое-то поле отсутствует

### 5.4. Создание РКО
**Метод:** `POST`  
**Endpoint:** `/cash-expense-orders`

**Параметры передаются:** через query string, не через JSON body

**Параметры:**
- `comment`
- `sum`

Пример:
```http
POST /cash-expense-orders?comment=изолента&sum=1223
```

**Важно:**
- не использовать JSON body
- параметры передавать через `params=`
- `comment` должен корректно URL-encoded
- `sum` передавать строкой в нормализованном виде

Логика кода:
```python
await client.post(
    "/cash-expense-orders",
    params={
        "comment": comment,
        "sum": str(sum_value)
    }
)
```

## 6. Поведение Telegram-бота

### 6.1. Команда `/start`
При вызове `/start` бот должен:
1. Проверить, что пользователь входит в whitelist.
2. Если доступ разрешён — показать приветствие.
3. Показать клавиатуру:
   - `Показать РКО`
   - `Создать РКО`
   - `Отмена`

### 6.2. Кнопка «Показать РКО»
При нажатии:
1. Бот вызывает `GET /cash-expense-orders?limit={DEFAULT_LIMIT}`
2. Получает JSON и читает массив `items`
3. Если `items` пустой — показывает сообщение: `РКО не найдены`
4. Если есть элементы — для **каждого** документа выводит отдельный блок в Telegram

**Формат вывода** — тот же, что задан в **разделе 5.3** (это единственный канонический формат для списка РКО):

```text
✅ TY000000017
Дата: 29.03.2026 0:19:42
Сумма: 1223
Касса: Касса нал
Статья: Прочие
Контрагент: —
Комментарий: тратата
```

Правила отображения — как в **5.3**:
- первая строка блока: `✅` или `❌` в зависимости от `posted`, затем номер документа (`number`)
- далее поля: дата, сумма, касса, статья затрат, контрагент, комментарий — по данным ответа 1С
- пустые `counterparty` и `comment` отображать как `—`
- порядок документов — как в ответе 1С
- отсутствующие поля не должны приводить к падению бота; нужен мягкий адаптер/маппер ответа 1С

### 6.3. Кнопка «Создать РКО»
Пошаговый сценарий:
1. Пользователь нажимает `Создать РКО`
2. Бот пишет: `Введите сумму и комментарий одной строкой, например: 1000 изолента`
3. Пользователь вводит строку
4. Бот парсит сумму и комментарий
5. Если сумма невалидна — бот пишет ошибку и просит ввести заново
6. Если комментарий пустой — бот отдельно запрашивает комментарий
7. Когда сумма и комментарий получены — бот показывает подтверждение: `Создать расход на 1000 с комментарием "изолента"?`
8. Кнопки:
   - `Подтвердить`
   - `Отмена`
9. Только после подтверждения вызывается POST в 1С
10. После успешного создания бот показывает ответ 1С и возвращает в главное меню

### 6.4. Быстрое создание по свободному сообщению
Если пользователь в обычном чате отправил:
- `1000 изолента`
- `1000,50 изолента`
- `1000.50 изолента`

бот должен:
1. распознать сумму
2. распознать комментарий
3. показать подтверждение
4. после подтверждения создать документ

## 7. Валидация суммы
Использовать `Decimal`.

**Допустимые форматы:**
- `1000`
- `1000.50`
- `1000,50`

**Недопустимые:**
- `0`
- отрицательные значения
- `abc`
- пустая строка

**Сообщение об ошибке:**
```text
Введите сумму числом, например: 1000 изолента
```

**Правила:**
- запятая должна преобразовываться в точку
- сумма должна быть строго больше 0
- комментарий — это весь оставшийся текст после суммы

## 8. Управление состояниями
Нужно использовать FSM.

### Минимальные состояния:
- ожидание ввода суммы и комментария
- ожидание отдельного комментария
- ожидание подтверждения

### Кнопка `Отмена`
Должна работать из любого состояния:
- сбрасывает текущее состояние
- возвращает пользователя в главное меню

## 9. Ограничение доступа
Бот не должен быть публичным.

Нужно реализовать whitelist по `Telegram user_id`.

### В env:
```text
TELEGRAM_ALLOWED_USER_IDS=123456789,987654321
```

Если пользователь не из списка, бот отвечает:
```text
У вас нет доступа к этому боту.
```

## 10. Webhook и HTTP-часть
Использовать **webhook**, не polling.

### Требования:
- FastAPI endpoint для Telegram webhook
- endpoint `/health`
- webhook path с секретной частью
- поддержка `secret token`, если используется
- регистрация webhook после деплоя
- возможность удалить webhook

### Пример:
```text
POST /telegram/webhook/<secret-path>
GET /health
```

## 17. Приёмочные тест-кейсы
1. `/start` показывает меню
2. `Показать РКО` вызывает GET endpoint 1С
3. При пустом списке выводится `РКО не найдены`
4. `1000 изолента` корректно распознаётся
5. `1000,50 изолента` корректно распознаётся
6. `1000.50 изолента` корректно распознаётся
7. `abc изолента` даёт ошибку
8. `0 изолента` даёт ошибку
9. `1000` без текста запрашивает комментарий отдельно
10. Без подтверждения документ не создаётся
11. После подтверждения вызывается: `POST /cash-expense-orders?comment=...&sum=...`
12. Ответ 1С после создания показывается пользователю
13. `Отмена` сбрасывает текущее состояние
14. Пользователь не из whitelist не получает доступ
